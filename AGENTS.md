# Marshall Agent Notes

Marshall is a permissionless distributed training network for small language models.

## Deployment Domains

- Use `marshall.training` for the public training network surface: coordinator console, active jobs, worker onboarding, and participation instructions.
- Use `marshall.chat` for chat/inference demos after model quality and serving are validated. The current milestone is a P2P prototype: Mac Pro HTTP gateway to explicit libp2p inference workers, with Caddy on the GCP VM reverse-proxying `marshall.chat` to VM-local `127.0.0.1:8787`. Until `marshall.chat` DNS points at the VM, keep `https://marshall.training/chat/` as the public HTTPS fallback path to the same gateway. Public TLS for `marshall.chat` requires `marshall.chat` DNS to point at static IP `34.148.63.131`.
- Use `./scripts/deploy-gcp-micro.sh` for the current public trial deployment. It targets two dedicated GCP VMs, keeps Redis local to each VM, runs one coordinator per VM on `0.0.0.0:8080`, exposes Caddy on public `80/443` from the primary only, exposes libp2p control peers on public TCP `4001` and `4002`, and stores generated admin secrets under ignored `.marshall/secrets/`.
- Use `./scripts/run-chat-gateway-local.sh` and `./scripts/run-chat-tunnel-gcp.sh` for the current Mac Pro hosted `marshall.chat` prototype instead of hand-written tmux commands. Both read ignored local config from `.marshall/secrets/chat-gateway.env` by default. On macOS, `./scripts/install-macos-chat-launchd.sh` installs persistent user LaunchAgents for both processes and `./scripts/uninstall-macos-chat-launchd.sh` removes them. The tunnel script must set `MARSHALL_CLOUDSDK_PYTHON` in launchd environments so gcloud uses Python 3.10+ instead of the system Python.
- Current public trial hosts: GCP project `iconic-elevator-394020`, zone `us-east1-b`, machine type `e2-small`. Primary is `marshall-micro-1` with static address `marshall-training-ip` (`34.148.63.131`) and internal IP `10.142.0.2`; replica is `marshall-micro-2` with static address `marshall-training-replica-ip` (`34.139.113.58`) and internal IP `10.142.0.3`. Redis must remain local to each VM on `127.0.0.1:6379`; never expose Redis publicly or share one Redis instance across nodes. Keep a 2GB local swapfile enabled for coordinator/control bursts. Workers join permissionlessly through `/control-network.json`; do not require or document a public `MARSHALL_SWARM_TOKEN`.

## Architecture Direction

- Use libp2p from the first implementation milestone.
- The coordinator is a control peer, not only an HTTP service.
- Workers are libp2p peers with persistent Ed25519 identities.
- Worker registration, job claim, job status, heartbeat, and artifact manifests must work over libp2p streams. Heartbeats may carry live progress telemetry (`progress_percent`, `progress_label`, `work_units_done`, `work_units_total`, `throughput_units_per_second`, `throughput_label`) so the dashboard can show per-worker progress and aggregate cluster speed without waiting for final artifacts.
- HTTP endpoints are admin/debug/dashboard conveniences, not the primary worker network.
- The public deployment uses federated active-active coordinators on separate machines: `public-primary` on `marshall-micro-1:8080` backed by local Redis `127.0.0.1:6379`, and `public-replica-1` on `marshall-micro-2:8080` backed by local Redis `127.0.0.1:6379`. `MARSHALL_COORDINATOR_PEERS` connects the stores over the private GCP network only. Jobs and artifacts are deterministically owner-sharded by id, writes route to the owner coordinator, and dashboard/API reads aggregate both stores with dedupe. This is independent multi-node Redis federation, not byzantine consensus.
- Target public participation: unknown workers may onboard, but trusted work requires validator verdicts, reputation, and coordinator-enforced suspension for bad actors.
- `ROADMAP.md` is the product/architecture roadmap for future larger-parameter fine-tuning, replicated inference, and trusted cluster modes. Keep public permissionless swarm work separate from trusted model-parallel cluster work.

## Initial Stack

- Node.js 22+ runtime for current js-libp2p dependencies.
- TypeScript libp2p for the p2p substrate.
- Python stdlib toy training runner for lightweight end-to-end smoke tests.
- Python MLX-LM runner for Apple Silicon LoRA jobs.
- Go for the native coordinator daemon.
- Redis for coordinator state and append-only event logs.
- Local filesystem artifacts for MVP storage.

## Implemented Prototype

- `src/node.ts` creates TCP + Noise + Yamux libp2p nodes with optional bootstrap and mDNS discovery.
- `src/identity.ts` persists Ed25519 private keys on disk.
- `src/control-cli.ts` starts a long-running configurable libp2p control peer.
- `src/control-cli.ts` supports `MARSHALL_JOB_COUNT` / `--job-count`; multi-job adapter runs use deterministic dataset shard jobs.
- `src/control-cli.ts` supports `--jobs-file` / `MARSHALL_JOBS_FILE` for mixed Marshall job definitions, including `evaluate_adapter` and `validate_artifact` jobs generated from coordinator artifacts.
- `src/control-cli.ts` supports `--coordinator-jobs true` / `MARSHALL_COORDINATOR_JOBS=true`. In this mode the control peer does not serve a static jobs file and does not initialize default jobs; it reads queued job specs live from the coordinator on every claim. The public VM should use this mode so train, eval, and validation phases do not require manual control-peer restarts.
- `src/control-cli.ts` and `src/dataset-run-cli.ts` can attach `resource_requirements.min_memory_gb` to generated jobs. Public adapter training jobs must declare `--min-memory-gb`; do not publish large-model jobs without a memory gate.
- `src/control-cli.ts` supports `--artifact-store-dir`, `--artifact-serve-dirs`, `--artifact-chunk-bytes`, and `--artifact-chunk-retries` for chunked p2p artifact payload transfer.
- `src/control-cli.ts` supports `--info-file`, `--public-control-host`, `--public-control-port`, `--public-control-addr`, `--coordinator-id`, and `--coordinator-role`; the public deployment uses this to publish each node's local `/control.json` and the primary-served `/control-network.json` for permissionless worker multiaddrs.
- `src/control-network-cli.ts` builds `/control-network.json` from multiple control peer info files. Workers should prefer this manifest over a single `/control.json` when joining the public network.
- The public `/control-network.json` should list `public-primary` on TCP `4001` at `marshall.training` and `public-replica-1` on TCP `4002` at the replica static IP. The primary Caddy serves `/control-replica.json` for the replica peer metadata and keeps `/control-mirror.json` as a legacy alias. The replica control peer must be connected to the replica coordinator so workers can claim, update, and publish through a separate Go coordinator process and a separate Redis-backed state store. `coordinator_url` values in the control manifest are local to each control process; public workers should use the control multiaddrs, not direct Redis or private coordinator URLs.
- `src/worker-cli.ts` starts a one-job worker that registers, claims, runs, publishes an artifact, and exits.
- `src/worker-cli.ts` accepts comma-separated control multiaddrs through `--control`, extra addresses through `--control-addrs` / `MARSHALL_CONTROL_ADDRS`, or a network manifest through `--control-network-url/path` / `MARSHALL_CONTROL_NETWORK_URL/PATH`; `src/worker-peer.ts` tries them in order, remembers the first successful address, and falls back when a dial or artifact provider fails.
- `src/worker-cli.ts` requires explicit `--memory-gb` / `MARSHALL_MEMORY_GB`. Do not let workers self-register with implicit memory values.
- Remote worker runs should receive every usable control multiaddr. If a LAN multiaddr times out while basic TCP probes succeed, check host firewall/routing and keep a second route or relay address available.
- `src/worker-cli.ts` materializes `marshall-artifact://<job_id>` job inputs from the control peer before `evaluate_adapter` or `validate_artifact` execution.
- `src/worker-cli.ts` retries transient artifact manifest publication failures, including empty p2p stream payloads during control peer restarts, before marking a completed local run as failed.
- `src/worker-pool-cli.ts` starts persistent concurrent worker slots against one or more control peers. Each slot runs a one-job worker process, then immediately claims more compatible work. Without `--max-jobs`, the pool keeps polling when the queue is empty; use `--max-jobs` and `--exit-when-idle` only for bounded tests or maintenance runs. Use this for product E2E proof instead of manual shell loops.
- `src/worker-supervisor-cli.ts` starts one persistent `model` worker pool that supports `train_adapter`, `evaluate_adapter`, and `validate_artifact` together. It can join through `--control-network-url/path` so a Mac keeps working if one public control peer is unavailable. Use this for public multi-Mac participation so a Mac can remain available across the whole model-production pipeline. Inference workers are a separate future role because they serve selected model packages and must keep models hot.
- Validation quorum requires enough distinct compatible worker identities. For `quorum=2`, an artifact produced or evaluated by one worker slot needs at least two other slots to vote because `src/control-peer.ts` rejects self-validation, same-peer validation, repeated validator IDs, and repeated local slot alternation keys. If a single Mac is doing the whole run, add enough low-memory worker slots for validation; memory gates should keep those slots away from high-memory training/evaluation jobs.
- Worker identity is the pair of `worker_id` and libp2p peer key. If a run reuses the same `--worker-id-prefix`, it must reuse the same `--key-dir`; otherwise the coordinator rejects registration with a peer mismatch to prevent identity takeover.
- `src/artifact-transfer.ts` implements chunked artifact bundles. Every chunk carries a SHA-256 hash, receivers retry corrupted chunks, every file is hashed after download, and the final artifact root hash must match the manifest before the control peer publishes it.
- `src/wire.ts` uses `MARSHALL_P2P_REQUEST_TIMEOUT_MS` for libp2p JSON request timeouts and defaults to 30 seconds so cold remote dials are not rejected by the old 5 second local-only timeout. It also provides JSONL stream helpers for inference stream events.
- `src/jobs.ts` defines local `train_toy_model`, `train_mlx_smoke`, and `train_adapter` job builders.
- `src/control-peer.ts` implements the in-memory control peer and handlers for worker registration, heartbeat, job claim, job status, artifact manifests, and artifact fetch. With `artifactStoreDir`, the control peer downloads worker artifact payloads over `/marshall/artifact/fetch/1.0.0`, stores verified local copies, rewrites artifact URIs to coordinator-accessible storage, and can serve stored artifacts to later workers.
- `src/control-peer.ts` rejects claims for jobs whose `resource_requirements.min_memory_gb` exceeds the registered worker memory before it asks the coordinator to mark the job claimed.
- `src/coordinator-client.ts` lets the TypeScript control peer persist lifecycle events into the Go coordinator over HTTP when `coordinatorUrl` is configured, sends the full `MarshallJob` as `job_spec`, and can read persisted jobs/artifacts back from the coordinator.
- `src/coordinator-client.ts` supports coordinator write authentication with `MARSHALL_COORDINATOR_TOKEN` / `--coordinator-token` and worker heartbeat forwarding for live coordinator leases.
- `src/coordinator-client.ts` can record artifact verdicts and read worker reputation through the coordinator API. Artifact verdict responses include `finalized`, `quorum`, `votes`, per-verdict `tally`, and `validator_reputations` when finalization scores validators; artifact records include `verdict_status`, `verdict_votes`, `verdict_quorum`, and `verdict_validators`.
- `src/coordinator-client.ts` can list coordinator artifacts; `src/validation-jobs-cli.ts` turns unvalidated target artifacts into distributed `validate_artifact` jobs, scopes targets with `--target-job-prefix`, and defaults to quorum 2 with two validator jobs per artifact.
- `coordinator/dashboard.go` hides worker rows after 15 minutes without recent activity. This is a dashboard visibility filter only: worker identity and reputation records stay in Redis.
- `src/worker-peer.ts` implements a worker peer that dials control peers and drives the first job lifecycle. It serves locally produced artifacts over `/marshall/artifact/fetch/1.0.0` while the manifest publish stream is open, and can fetch verified input artifacts back from any configured control peer. For artifact fetch, `accepted:false` responses such as `artifact not found` are treated as provider misses and the next control peer is tried.
- `src/control-peer.ts` rejects `evaluate_adapter` claims when the candidate evaluator produced the source adapter, uses the same libp2p peer, or has the same local worker slot suffix across train/eval worker IDs. It also rejects `validate_artifact` claims when the worker/peer/slot produced the target artifact, already voted for the target, or belongs to a slot that already voted. Producer, evaluator, and validator roles should remain separated wherever possible.
- The public control peer is permissionless. Do not add worker join tokens to public onboarding, examples, deploy scripts, dashboard copy, or `coordinator/public/AGENTS.md`.
- `src/training-runner.ts` runs the local toy trainer for `train_toy_model` jobs and validates the emitted manifest and metrics.
- `src/training-runner.ts` also wraps `training/mlx_linear_smoke.py` for `train_mlx_smoke` jobs and emits an `mlx_smoke_result` artifact manifest.
- `src/training-runner.ts` wraps `training/mlx_lora_smoke.py` for `train_adapter` jobs and emits a `lora_adapter` artifact manifest. MLX-LM must receive the content-addressed dataset cache directory, even when the shard materializes from a single JSONL file. The runner listens for `MARSHALL_PROGRESS {json}` lines and forwards them into worker heartbeat telemetry.
- `src/training-runner.ts` wraps `training/mlx_ag_news_eval.py` for `evaluate_adapter` jobs with `eval_kind: "ag_news"` and `training/mlx_lora_eval.py` for jobs with `eval_kind: "instruction_terms"`, then emits an `adapter_evaluation` artifact manifest.
- `src/training-runner.ts` runs `validate_artifact` jobs for validator workers. The current validator checks target artifact hash, adapter-evaluation schema, metric self-consistency, and policy thresholds, then emits an `artifact_validation` manifest with `accepted`, `poor`, `rejected`, or `malicious` plus the requested quorum. Internally inconsistent evaluation metrics are treated as malicious.
- `src/control-peer.ts` forwards `artifact_validation` manifests into coordinator validator votes, so target worker reputation is updated only after a verdict reaches coordinator quorum.
- `src/evaluation-jobs-cli.ts` scans `lora_adapter` manifests and creates `evaluate_adapter` jobs for a held-out eval shard. It requires explicit `--artifact-uri-mode`, `--eval-kind`, `--run-id`, `--round-id`, `--job-prefix`, `--model`, `--max-examples`, and `--max-tokens`; do not add implicit defaults for evaluation semantics. Use `--eval-uri` / `MARSHALL_EVAL_URI` when the coordinator hashes a local eval file but remote workers need a different worker-resolvable URI. Use `--artifact-uri-mode p2p` to emit `marshall-artifact://<job_id>` adapter references for remote workers. Use `--min-memory-gb` or `--eval-min-memory-gb` when evaluation loads a memory-sensitive base model.
- `src/validation-jobs-cli.ts` supports `--target-uri-mode p2p`, so validators fetch target artifacts from the control peer instead of reading coordinator-local `file://` paths.
- `src/validation-top-up.ts` creates additional `validate_artifact` jobs for adapter-evaluation artifacts whose verdict is still not finalized after existing validation jobs are terminal. It preserves the original target suffix when possible, increments the next `vote_NNN`, skips existing job IDs, and emits tie-break votes when validators disagree without reaching quorum.
- `src/round-orchestrator-cli.ts` is the product path for advancing one run phase. It reads coordinator artifacts or an artifact snapshot and can schedule evaluation jobs from LoRA artifacts, schedule validation jobs from unvalidated adapter-evaluation artifacts, or produce accepted-only leaderboard/model-package outputs after validation finalizes. When a control source is provided, final selection fetches evaluation and adapter artifacts through the libp2p control network with chunked hash verification instead of assuming the selected artifact exists on the primary node's filesystem.
- `src/round-daemon-cli.ts` is the product path for unattended round advancement. It polls coordinator jobs/artifacts, waits for train/eval/validation phase completion, writes filtered artifact snapshots for the current run, invokes the round orchestrator, publishes validation top-up jobs when quorum votes are missing or tied, and stops after model selection. If the state file already contains `action: "selected"` for the same run and round, it exits with `already_selected` and must not republish model packages on restart. Use it instead of manually switching the public control peer or hand-running phase commands.
- `src/model-selection.ts` centralizes adapter ranking. The current policy is `best_adapter_by_eval_score`, score `accuracy - invalid_rate`, deterministic tie breakers, accepted-only filtering when requested, and `merge_mode: single_adapter`.
- `src/leaderboard-cli.ts` scans artifact stores for `adapter_evaluation` metrics, skips unrelated training/validation metrics, and writes `leaderboard.json`, `top_k.json`, and `optimized_model.json` with an explicit `selection_policy`. With `--coordinator-url --require-verdict accepted`, it filters model selection to coordinator-accepted artifacts only.
- `src/model-package-cli.ts` packages or republishes the selected optimized model as base model + LoRA adapter metadata, writes an `optimized_model_package` manifest, copies a full control-readable manifest into the control artifact store, and updates the model registry. Use `--adapter-artifacts-dir` so packages point at the control peer's verified local adapter copy instead of a worker-local path embedded in evaluation metrics. Model payloads must not be downloaded over HTTPS; HTTPS may expose only metadata such as `/models/index.json`.
- `src/model-package.ts` owns model registry parsing/loading/selection. Use the registry as metadata-only source of truth for ready model packages; never expose package or adapter payloads through HTTPS.
- `src/model-promote-cli.ts` is the manual worker-side promotion path for ready models. It requires an explicit control source (`--control`, `--control-addrs`, or `--control-network-url/path`), `--package-job-id`, and `--package-hash`, fetches the package and adapter from the control peer via `/marshall/artifact/fetch/1.0.0`, verifies every chunk/file/root hash through `src/artifact-transfer.ts`, then writes a local `model_package.json` whose `adapter_path` points at the fetched local adapter. Prefer inference-worker registry auto-cache for product flows; keep this CLI for debugging, offline inspection, and explicit recovery.
- `src/model-query-cli.ts` queries a packaged optimized model against a selected eval record and can fail unless the answer is correct.
- `training/mlx_lora_chat.py` supports normal JSON completion output and `--stream-jsonl`. Streaming mode forwards real stdout chunks from `mlx_lm.generate` as `marshall_inference_stream_event` JSONL and emits a final cleaned `completed` event.
- `src/inference-worker-cli.ts` starts a libp2p inference worker for `marshall.chat`. The worker owns the base model + adapter paths, handles `/marshall/inference/hello/1.0.0` capability probes, `/marshall/inference/generate/1.0.0`, and `/marshall/inference/generate_stream/1.0.0`, and runs `training/mlx_lora_chat.py` locally to return generation responses or stream events. If no local `--model-package` / `MARSHALL_MODEL_PACKAGE` is provided, it may read `--model-registry-url` or `--model-registry-path`, select an explicit `--model-package-id` or exact model/adapter selector, fetch the package and adapter from an explicit control source (`--control`, `--control-addrs`, or `--control-network-url/path`) via libp2p artifact fetch, verify all hashes, cache the local model package, and only then start serving. If multiple ready registry models exist and no selector is provided, it must fail instead of choosing an implicit default.
- `src/inference-router.ts` is the gateway-side inference router. It probes configured worker multiaddrs, marks workers ready when they respond with capabilities, exposes non-sensitive worker snapshots, filters compatibility per request target (`model`, `adapter_id`, `adapter_hash`), selects the least-busy compatible worker, streams JSONL events when workers support `generate_stream`, and falls back across compatible workers when generation fails.
- `src/chat-memory.ts` implements durable gateway-owned chat memory with file-backed `conversation_id` records under `.marshall/chat/conversations` by default. This memory includes message history, model package metadata, and structured long-term memory: `summary`, `facts`, `preferences`, `goals`, `open_tasks`, and `plans`. Inference workers must remain stateless and must not store private chat history or long-term user memory.
- `src/chat-server-cli.ts` serves the `marshall.chat` gateway. In `--runtime p2p_worker` mode it serves `chat/public`, accepts `--p2p-worker-addr` plus comma-separated `--p2p-worker-addrs`, exposes `GET /api/health`, `GET /api/models`, `GET /api/inference/workers`, `GET /api/conversation?conversation_id=<id>`, `POST /api/conversation/memory`, `POST /api/chat`, and SSE `POST /api/chat/stream`, persists conversation turns, builds a bounded context window, and forwards prompts through the inference router. `GET /api/models` may read a local registry file or metadata URL but must not proxy model payloads. `POST /api/chat` and `POST /api/chat/stream` may carry `model_package_id`; the gateway resolves it through the registry and routes to compatible workers. Use this for real distributed inference tests. `local_process` mode remains available only for single-machine debugging.
- Long-term memory is included in worker prompts only as a bounded gateway-composed context block. Use `--max-context-messages` for recent turns and `--max-memory-items` for long-term memory items. Do not move memory state into inference workers; sticky routing is an optimization only. Do not expose the gateway-composed prompt in public SSE events, logs, dashboards, or worker lists; `completed` stream events must omit internal prompt/memory context.
- `chat/public/app.js` submits the composer on `Enter`, preserves multiline input on `Shift+Enter`, uses `POST /api/chat/stream` for streamed answers, lets the user edit persisted long-term memory/plans, lets the user select a ready model package by `model_package_id`, and displays inference worker readiness, ready model packages, and the last selected worker without exposing raw worker multiaddrs or long hashes in visible event text.
- `scripts/check-chat-public.sh` is the readiness gate for public chat inference. It requires explicit `--url` and `--expected-ip`, verifies DNS, pins `curl` to the expected IP to avoid stale local resolver caches, verifies HTTPS health, one streamed P2P response, at least one ready worker, and checks that public `completed` SSE events do not leak the gateway-composed prompt. Run it against `https://marshall.training/chat/` for the fallback and `https://marshall.chat` before declaring the chat domain operational.
- `scripts/run-chat-gateway-local.sh` starts the compiled `marshall.chat` gateway from an ignored env file, validates Node.js 22+, fails when required P2P/model/adapter values are missing, and passes optional `MARSHALL_MODEL_REGISTRY_PATH` / `MARSHALL_MODEL_REGISTRY_URL` through to `GET /api/models`. `scripts/run-chat-tunnel-gcp.sh` starts the reverse SSH tunnel to the GCP VM and supports explicit `MARSHALL_CLOUDSDK_PYTHON` for launchd. `scripts/install-macos-chat-launchd.sh` writes user LaunchAgents for both processes with `KeepAlive`; logs go under `.marshall/logs`.
- `scripts/run-inference-worker-local.sh` starts the compiled inference worker from an ignored env file and requires explicit Node.js, libp2p key, listen multiaddr, stable worker id, and Python binary. It supports three serving modes: local `MARSHALL_MODEL_PACKAGE`, registry-backed P2P auto-cache through `MARSHALL_CONTROL_ADDR`/`MARSHALL_CONTROL_NETWORK_URL` + `MARSHALL_MODEL_REGISTRY_URL`/`PATH` + optional `MARSHALL_MODEL_PACKAGE_ID`, or explicit model/adapter metadata for debugging. `scripts/install-macos-inference-launchd.sh` and `scripts/uninstall-macos-inference-launchd.sh` manage a persistent macOS LaunchAgent for inference workers. Use these scripts for repeated Mac worker runs instead of manual `nohup` or `tmux`.
- `tests/chat-memory.test.ts` verifies persistent conversation storage, safe conversation IDs, and context trimming. `tests/chat-server.test.ts` verifies local chat streaming, persisted streamed conversations, and model registry metadata. `tests/inference-p2p.test.ts` verifies the distributed chat path with real libp2p peers, including worker capability probes, P2P streaming, router failover across compatible inference workers, and model-package-id routing across different worker capabilities. `tests/p2p.integration.test.ts` verifies chunked artifact fetch, model promotion over libp2p, and artifact-provider failover across multiple control peers. Run P2P tests with Node.js 22+ outside restricted sandboxes because they open TCP sockets.
- `src/e2e-ag-news-cli.ts` runs the AG News product E2E path: training worker pool, p2p artifact upload to the control store, p2p adapter download for evaluation workers, evaluation artifact upload, optional p2p validation target download, accepted-only leaderboard selection, package from verified adapter storage, query, and optional coordinator persistence verification.
- `src/dataset-manifest-cli.ts` builds private content-addressed dataset manifests from local JSONL inputs under `.marshall/`, with optional external `--base-uri` shard URLs for remote workers. It accepts existing `messages[]` chat records, `--text-field` plain text records, and instruction-tuning records through `--instruction-field`, `--response-field`, optional `--context-field`, and optional `--system-prompt`.
- `src/dataset-run-cli.ts` is the product path for dataset-backed training work. It can build the manifest, generate the run bundle, write `jobs/train-adapters.json`, and publish those jobs to the coordinator when `--coordinator-url` is provided.
- `DatasetShard.files[]` is the dataset transfer contract for remote workers: each entry carries `path`, worker-resolvable `uri`, `sha256`, and optional `bytes`; the shard hash still commits to the assembled shard directory.
- `TrainingJob.training_config` is the adapter training contract. For `train_adapter` jobs it must carry the base model and MLX LoRA hyperparameters (`model`, `iters`, `batch_size`, `learning_rate`, `num_layers`, `max_seq_length`, report/eval cadence, seed, prompt masking, and gradient checkpointing). Workers must prefer this job spec over local environment overrides.
- `training/tiny_char_lm.py` trains a tiny character bigram language model with stdlib-only SGD and writes `model.json`, `metrics.json`, `train.log`, and `manifest.json`.
- `training/mlx_linear_smoke.py` verifies MLX GPU execution with a tiny gradient-descent job on Apple Silicon.
- `training/mlx_lora_smoke.py` runs a tiny MLX-LM LoRA training job, streams the child `mlx_lm.lora` process into log files, emits `MARSHALL_PROGRESS` JSON lines from observed iteration output, writes `metrics.json`, captures train/validation loss, and validates adapter files.
- `training/build_marshall_instruction_dataset.py` generates and validates deterministic train/valid/test/eval splits for Marshall coordinator-event tasks under `.marshall/datasets/marshall-instructions`.
- `training/mlx_lora_eval.py` runs held-out generation checks against a base model or LoRA adapter and writes eval metrics.
- `training/build_ag_news_dataset.py` downloads AG News CSVs into `.marshall/cache/raw/ag-news`, builds local train/valid/test/eval JSONL plus 4 shards under `.marshall/datasets/ag-news`, and writes a manifest consumed by adapter job creation.

- `training/mlx_ag_news_eval.py` evaluates base models or LoRA adapters on AG News exact-label accuracy.
- `src/dataset-cache.ts` materializes assigned dataset shards into a content-addressed local cache and verifies hashes before training or evaluation. Single JSONL eval shards remain addressable as files after caching.
- `src/dataset-cache.ts` supports shard-level `files[]` manifests. Workers download or copy only assigned shard files, verify each file SHA-256 and optional byte size, then verify the final shard root hash before training.

Conversation memory is intentionally not part of the inference worker schema; the gateway composes `InferenceRequest.prompt` from persisted private conversation state.

The control CLI exposes this as `--adapter-dataset ag_news|manifest` or `MARSHALL_ADAPTER_DATASET=ag_news|manifest`, with `--adapter-dataset-dir` / `MARSHALL_ADAPTER_DATASET_DIR` pointing at the local manifest directory.

- `cmd/marshall-coordinator` is the native Go coordinator entry point. It reads `MARSHALL_COORDINATOR_ID` and `GET /health` returns that id so active coordinator replicas can be verified independently.
- `cmd/marshall-coordinator` reads `MARSHALL_COORDINATOR_PEERS` as comma-separated `id=url` entries. When configured, the process wraps its local Redis store in `FederatedStore`, routes owner-sharded writes to the correct coordinator, and aggregates workers/jobs/artifacts/events across peers. Federation peer HTTP calls use `X-Marshall-Federation: 1` so peer handlers execute local-only operations and do not recurse.
- `coordinator/federated_store.go` aggregates worker reputation across peer stores before accepting claims. Reputation counters are summed and score/status are recomputed pessimistically, so a worker suspended on one shard cannot claim through another coordinator.
- `coordinator/redis_store.go` stores runs, workers, jobs, full job specs, job claims, statuses, artifacts, validator vote hashes, and append-only events in Redis. `GET /artifacts` must hydrate `verdict_validators` with batched Redis reads, not one Redis round trip per artifact.
- `coordinator/redis_store.go` treats repeated job publication as an idempotent upsert: restarts may refresh metadata/spec but must not reset status, worker assignment, or lifecycle progress.
- `coordinator/redis_store.go` maintains job leases and can requeue expired running jobs so abandoned work is visible and recoverable.
- `coordinator/redis_store.go` tracks worker reputation and blocks suspended workers from claiming more jobs. Artifact verdicts with `quorum > 1` are stored as per-validator votes first and finalize only when one verdict reaches quorum. Quorum finalization is protected by a per-artifact Redis `SET NX` lock so concurrent validator votes cannot apply duplicate reputation deltas. Validators must be registered active workers that support `validate_artifact` and cannot validate their own artifacts. When a verdict finalizes, validator votes aligned with the final verdict are rewarded; validator votes that diverge are penalized, including hard suspension for accepting an artifact that finalizes as `malicious` or falsely marking an accepted artifact as `malicious`. Verdict policy is currently `accepted +2`, `poor -10`, `rejected -25`, `timeout -15`, and `malicious -100`, capped to the `0..100` score range.
- `coordinator/http.go` exposes the coordinator HTTP admin API, including `GET /health`, `GET /jobs`, `GET /jobs/{job_id}`, `GET /artifacts`, `GET /artifacts/{job_id}`, `POST /artifacts/{job_id}/verdict`, `GET /workers/{worker_id}/reputation`, `POST /jobs/requeue-expired`, `GET /dashboard`, `GET /events/stream`, and the embedded public console at `/`.

- The dashboard shows worker status, job status, active-run completion, per-worker progress, cluster speed, artifact verdicts, and live coordinator events. The large completion percentage must be phase-weighted across the full train/evaluate/validate pipeline: training, evaluation, and validation each contribute one third. A phase with no published jobs is still displayed as not scheduled, but it contributes 0% to the large E2E completion percentage so training-only completion can never show as 100%. The event stream must display compact identifiers only and avoid full worker IDs, full peer IDs, raw messages, or other sensitive/debug-heavy payloads.
- The dashboard groups coordinator jobs by canonical run. Active runs drive the top contract, completion panel, and jobs table by default; completed historical runs stay hidden until the user clicks `show history`. The active-run completion panel belongs above `past runs`. The jobs table must paginate visible rows in pages of 6, and workers plus event stream should remain a 50/50 live grid on desktop.

## Development Rules

- Keep all docs in English.
- Do not add, remove, or publish repository examples, sample datasets, demo datasets, or local test harness artifacts unless the user explicitly asks for that specific repo content. Keep exploratory fixtures, adversary runs, generated datasets, and one-off validation artifacts outside the repo, preferably under `.marshall/` or `/tmp`.
- Build and test Marshall as a product, not as manual orchestration. Do not replace missing product behavior with shell loops, ad hoc scripts, or the agent acting as the coordinator.
- E2E validation must go through Marshall contracts: coordinator creates jobs, workers claim and run them, artifacts are published, evaluation jobs are scheduled, leaderboard/top-K is derived from artifacts, and only then merge/selection runs.
- Artifact manifests and artifact payload bytes must travel through Marshall protocols for product validation. Do not use manual `rsync` or local file copy as product behavior.
- Artifact payload transfer must stay chunked for large files. Every chunk must be hash-checked, failed chunks must be retried, every completed file must be hash-checked, and the final artifact root hash must match the manifest before publication or downstream use.
- Remote worker jobs must use dataset and artifact URIs resolvable from the worker machine. If coordinator-local and worker-local paths differ, emit worker-resolvable URIs in job specs while preserving coordinator-side hash verification.
- Dataset manifests for product tests must describe real assigned shard bytes, not placeholder work. Workers should download only their assigned shard files and must fail on hash or size mismatch.
- Manual shell commands are allowed only to start/stop services, inspect state, or run tests. If a repeated/manual operational step is needed to prove the system, implement it as Marshall code or documented CLI behavior first.
- Do not add blockchain, token, payment, or incentive code before validation and reputation work.
- Do not build synchronous distributed training.
- Do not assume all workers are equally capable.
- Runtime-test networking changes with at least one control peer and one worker peer before marking them done.
- For backend/API functions, add integration tests.
- Runtime-test coordinator storage/API changes against real Redis before pushing.
